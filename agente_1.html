Simulación del Loop de Razonamiento

El loop empieza cuando el agente recibe una tarea en lenguaje natural. El historial de mensajes arranca vacío.
messages[ ]iter 0
user
Encontrá todos los pedidos de más de $500 del mes pasado y enviá un resumen por email a ventas@corp.com
Evalúa qué tools disponibles puede necesitar.
El LLM procesa el task y decide que necesita información. Genera un bloque tool_use.
messages[ ]LLM pensando…
user
Encontrá todos los pedidos...
assistant — razonamiento interno
Para completar esta tarea necesito consultar los pedidos. Voy a llamar a query_orders primero.
stop_reason = "tool_use" — necesita datos externos.
El agente extrae el bloque, pasa los parámetros por FastMCP, y ejecuta la tool real.
messages[ ]ejecutando tool…
tool_use → query_orders
{ "min_amount": 500, "date_from": "2026-04-01", "date_to": "2026-04-30" }
FastMCP valida params → ejecuta query.
El resultado se agrega al historial. El loop vuelve a empezar con más contexto.
messages[ ]↻ volviendo al LLM
tool_result
[ { "id": "ord_001", "amount": 820, "client": "Acme SA" }, { "id": "ord_002", "amount": 1340, "client": "Beta SRL" } ]
Todo el historial viaja al LLM de nuevo.
Con los datos en el historial, el LLM genera una segunda tool call (email).
messages[ ]segunda tool…
tool_use → send_email
{ "to": "ventas@corp.com", "subject": "Pedidos >$500", "body": "Se encontraron 2 pedidos. Total: $2160" }
FastMCP valida destinatario → envío.
El resultado del email entra al historial. El LLM concluye que la tarea está cumplida.
messages[ ]end_turn — loop completo
tool_result — send_email
{ "status": "sent", "message_id": "msg_x7k2p9" }
assistant — respuesta final
Listo. Encontré los pedidos y envié el resumen a ventas@corp.com.
stop_reason = "end_turn" — termina la iteración.

Casos de Uso: REST vs FastMCP

REST — el cliente es un humano o app
📱
App móvil o web — El usuario toca un botón, la app llama a un endpoint específico conocido.
🔄
Integración B2B — Sistemas externos que consumen tu API pública con contratos fijos.
FastMCP — el cliente es un LLM
🧠
Agente autónomo — El modelo decide qué herramienta usar, cuándo, y con qué parámetros.
s🔀
Flujos emergentes — La secuencia de llamadas emerge del razonamiento, no de código hardcodeado.

Contrato Derivado de Código

Comparación entre documentar contratos a mano (OpenAPI/Swagger) y derivarlos automáticamente desde el código (Pydantic/FastMCP).

REST Tradicional (Drift)
1. Cambio de requisito: mínimo de 30 a 60 días
# openapi.yaml minimum: 60 ← actualizaste # handler.py (Lógica Backend) if days < 30: ← olvidaste raise ...
Código desincronizado
FastMCP (Single Source of Truth)
Cambio de requisito: mínimo de 30 a 60 días
class DeleteUserParams(BaseModel): days_inactive: int = Field( ..., ge=60, ← un solo cambio le=365 )
1 lugar — todo sincronizado

Lo que FastMCP genera automáticamente

1. Schema JSON (al LLM)
2. Input del Modelo
3. Rechazo de Validación
// Generado automáticamente. El LLM lee esto antes de actuar. { "name": "delete_user", "inputSchema": { "properties": { "days_inactive": { "type": "integer", "minimum": 60 } } } }
// El LLM intenta enviar un dato inválido (menor a 60) { "user_id": "usr_admin", "days_inactive": 5 }
# Error que FastMCP devuelve al LLM sin crashear el sistema ValidationError: 1 validation error days_inactive Input should be greater than or equal to 60 Input: 5 [type=greater_than_equal] # El modelo lee el error, se autocorrige y reintenta.
