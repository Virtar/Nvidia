Libro.cs  ·  Persona 1  ·  Concepto: Encapsulamiento
1
Encapsulamiento — atributos private ▶
Todos los campos (titulo, autor, editorial) son private. Ninguna clase externa puede modificarlos directamente, protegiendo la integridad del objeto.
// Nadie fuera de Libro puede hacer libro.titulo = "otro" private string titulo; private string autor; private string editorial;
¿Por qué importa? Si fueran public, cualquier parte del programa podría corromper los datos del libro sin que nadie se entere. El encapsulamiento es la primera barrera de calidad.
2
Constructor con parámetros obligatorios ▶
El constructor exige los tres datos. No existe forma de crear un Libro incompleto — el compilador lo rechaza en tiempo de compilación.
public Libro(string titulo, string autor, string editorial) { this.titulo = titulo; this.autor = autor; this.editorial = editorial; }
¿Por qué importa? Elimina una categoría entera de errores: objetos a mitad de inicializar. Un libro sin título no debería poder existir nunca.
3
Polimorfismo — override ToString() ▶
Se sobrescribe el método ToString() heredado de object. Permite mostrar un libro con Console.WriteLine(libro) sin código adicional.
public override string ToString() { return $"Título: {titulo} | Autor: {autor} | Editorial: {editorial}"; }
¿Por qué importa? Es una convención del lenguaje. Cualquier herramienta que imprima objetos usará esta versión automáticamente. Mejora la depuración y los tests.
Lector.cs  ·  Persona 2  ·  Concepto: Asociación + Colecciones
4
Asociación — List<Libro> en memoria ▶
El lector guarda objetos Libro completos, no solo IDs. Esto es una asociación real en memoria: el lector "posee" sus libros prestados.
// Asociación: el lector tiene libros prestados private List<Libro> librosPrestados;
¿Por qué importa? Con el objeto completo podemos acceder a título, autor, etc., sin buscar en otra estructura. Más eficiente y natural que guardar solo un ID.
5
Inicialización defensiva en el constructor ▶
La lista se inicializa en el constructor. Si quedara en null, cualquier llamada a .Add() lanzaría NullReferenceException.
public Lector(string nombre, int dni) { this.nombre = nombre; this.dni = dni; // Lista vacía, nunca nula. Lista para usar desde el primer momento. this.librosPrestados = new List<Libro>(); }
¿Por qué importa? Programación defensiva: el objeto nace listo para operar. Evita el error más común en OOP con colecciones: el NullReferenceException.
6
Propiedad calculada — CantidadLibros ▶
En vez de exponer la lista (que permitiría manipularla desde afuera), se expone solo el conteo. Biblioteca puede verificar el límite sin romper el encapsulamiento.
// Solo lectura. Delega en List.Count. public int CantidadLibros => librosPrestados.Count; // En Biblioteca: if (lector.CantidadLibros >= 3) return "TOPE DE PRESTAMO ALCANZADO";
¿Por qué importa? Si el límite cambia de 3 a 5, solo se modifica Biblioteca. Lector no cambia. Alta cohesión, bajo acoplamiento — pilares de calidad del software.
Biblioteca.cs  ·  Persona 3  ·  Concepto: Controlador + DRY + Fail Fast
7
Colecciones genéricas — List<T> ▶
Se usan List<Libro> y List<Lector> en lugar de ArrayList. El parámetro de tipo garantiza que solo entren los objetos correctos.
private List<Libro> libros = new List<Libro>(); private List<Lector> lectores = new List<Lector>();
¿Por qué importa? Type-safety en tiempo de compilación. Es imposible agregar un Lector a la lista de Libros. ArrayList permitiría eso y el error aparecería recién en tiempo de ejecución.
8
Principio DRY — métodos privados auxiliares ▶
La lógica de búsqueda está centralizada en métodos privados. No se repite Find() en cada método público que lo necesite.
// DRY: la lógica de búsqueda en un solo lugar private Libro buscarLibro (string t) => libros.Find(l => l.getTitulo() .Equals(t, StringComparison.OrdinalIgnoreCase)); private Lector buscarLector(int d) => lectores.Find(l => l.getDni() == d);
¿Por qué importa? DRY = Don't Repeat Yourself. Si cambia la lógica de búsqueda, se modifica en un solo lugar. Mantenimiento más simple y menos errores.
9
Validaciones jerárquicas — patrón Fail Fast ▶
Las tres validaciones están en orden deliberado. Si algo falla, se retorna inmediatamente sin trabajo innecesario.
// Orden: actor → su límite → el recurso if (lector == null) return "LECTOR INEXISTENTE"; if (lector.CantidadLibros >= 3) return "TOPE DE PRESTAMO ALCANZADO"; if (libro == null) return "LIBRO INEXISTENTE";
¿Por qué importa? Detectar errores lo antes posible ahorra trabajo y da mensajes precisos al usuario. Si el lector no existe, no tiene sentido buscar el libro.
10
Transferencia real del objeto en memoria ▶
El mismo objeto Libro se quita del stock y se entrega al lector. Modela fielmente la realidad: un libro prestado ya no está en la estantería.
// El libro sale del stock de la biblioteca... libros.Remove(libro); // ...y entra a la colección del lector. lector.AgregarLibroAPrestamo(libro);
¿Por qué importa? Sin el Remove(), el libro podría prestarse dos veces. La consistencia del estado del sistema es responsabilidad exclusiva de Biblioteca.
Flujo de prestarLibro() — 7 pasos con validaciones jerárquicas
1

Recibir parámetros: titulo (string) + dni (int)

El método público acepta el título del libro y el DNI del lector que quiere tomarlo prestado.

2

buscarLector(dni) → ¿existe en el sistema?

Se llama al método privado auxiliar. Recorre la lista con Find(). Devuelve null si no lo encuentra.

✗

Lector null → return "LECTOR INEXISTENTE"

Retorno inmediato. Fail Fast: no tiene sentido continuar si el actor no está registrado.

3

lector.CantidadLibros >= 3 → ¿alcanzó el tope?

Se consulta la propiedad calculada de Lector. Regla de negocio: máximo 3 préstamos simultáneos.

✗

Tope alcanzado → return "TOPE DE PRESTAMO ALCANZADO"

El lector existe pero ya tiene el máximo. No se busca el libro: más ahorro de trabajo.

4

buscarLibro(titulo) → ¿existe en el stock?

Busca en la lista con OrdinalIgnoreCase. "Clean Code" y "clean code" se tratan igual.

✗

Libro null → return "LIBRO INEXISTENTE"

No está en stock: ya fue prestado, no fue registrado, o hay un error de tipeo en el título.

✓

Transferencia → return "PRESTAMO EXITOSO"

libros.Remove(libro) lo saca del stock. lector.AgregarLibroAPrestamo(libro) lo entrega. El objeto cambia de mano.
