Loop de Razonamiento

Simulación paso a paso del ciclo interno de un agente LLM que encadena tool calls para resolver una tarea en lenguaje natural.

El loop empieza cuando el agente recibe una tarea en lenguaje natural. El historial de mensajes arranca vacío — solo existe el task del usuario. El LLM todavía no sabe qué herramientas va a necesitar.

user
Encontrá todos los pedidos de más de $500 del mes pasado y enviá un resumen por email a ventas@corp.com
El LLM recibe el task. Evalúa qué tools disponibles puede necesitar para resolverlo.
El historial es todo el contexto
El agente no tiene memoria propia. Cada llamada al LLM incluye todos los mensajes anteriores — user, assistant, tool results.
Tools disponibles
El servidor MCP le pasó al LLM el schema de query_orders y send_email. El modelo las conoce antes de empezar.
El LLM procesa el task y decide que necesita información antes de poder responder. No inventa los datos — razona que debe consultar primero. Genera un bloque tool_use en lugar de texto final.

assistant — razonamiento interno
Para completar esta tarea necesito: (1) consultar los pedidos del mes pasado filtrados por monto, (2) luego enviar el resumen. Voy a llamar a query_orders primero.
stop_reason = "tool_use" — el modelo no terminó, necesita datos externos.
El modelo planifica
Antes de llamar a la primera tool, el LLM ya sabe que va a necesitar dos pasos. Esa planificación implícita emerge del razonamiento.
stop_reason controla el loop
Si stop_reason == "tool_use" → seguir el loop. Si stop_reason == "end_turn" → terminar y devolver la respuesta.
El agente extrae el bloque tool_use de la respuesta del LLM, pasa los parámetros por FastMCP (validación Pydantic), ejecuta la tool real, y captura el resultado.

FastMCP valida params → ejecuta query → captura resultado.
El agente no ve SQL
La tool query_orders puede internamente correr una query en Postgres. El agente solo ve los parámetros y el resultado.
Checkpoint antes de ejecutar
Antes de correr una tool destructiva, el StateManager guarda un snapshot. Si la tool falla, el estado se revierte.
El resultado de la tool se agrega al historial como un mensaje tool_result. El historial completo se manda de vuelta al LLM. El loop vuelve a empezar — ahora con más contexto.

Todo el historial → LLM de nuevo. El modelo ahora tiene los datos y puede decidir el siguiente paso.
El contexto crece con cada iteración
El LLM en cada vuelta ve todo lo que pasó. Puede razonar sobre los resultados anteriores para decidir la próxima acción.
Sin memoria propia
El modelo no "recuerda" entre llamadas. Todo lo que sabe en esta vuelta viene del historial que el agente le pasa.
Con los datos de los pedidos en el historial, el LLM genera una segunda tool call — ahora para enviar el email. Construye el cuerpo del mensaje a partir de los resultados anteriores.

FastMCP valida destinatario contra whitelist → hook de auditoría → envío.
El LLM compone el contenido

El pre-hook de send_email verifica el destinatario contra la whitelist antes de enviar.
El resultado del email entra al historial. El LLM evalúa el contexto completo y concluye que la tarea está cumplida. Genera texto final — stop_reason = "end_turn". El loop termina.

El loop duró 2 iteraciones
Tarea simple → 2 tool calls. Una tarea más compleja puede necesitar 5, 10 o 20 iteraciones. El loop termina cuando el modelo decide que terminó.
Todo el historial persiste
El historial completo queda disponible para auditoría: qué consultó, qué datos obtuvo, qué envió, en qué orden.

Módulo 02

REST vs FastMCP

Cuándo usar cada protocolo. La clave está en quién decide qué llamar: un humano, código determinista, o el LLM.

REST — el cliente es un humano o app
App móvil o web — un usuario toca un botón, la app llama a POST /orders. El cliente sabe exactamente qué endpoint usar.
Integración entre sistemas — Shopify llama a tu webhook, Stripe te notifica un pago. Contratos fijos, conocidos de antemano.
API pública — terceros consumen tu servicio. Necesitás versionado, autenticación OAuth, documentación estable.
Polling y webhooks — el cliente decide cuándo consultar. El flujo lo controla el humano o el scheduler.
FastMCP — el cliente es un LLM
Agente autónomo — el modelo decide qué herramienta usar, cuándo, y con qué parámetros. No hay humano eligiendo el endpoint.
Asistente con acceso a sistemas — "borrá los usuarios inactivos de más de 90 días". El agente traduce lenguaje natural a tool calls.
Flujos multi-paso no predefinidos — el agente encadena tools según el contexto. La secuencia emerge del razonamiento.
Exploración de datos — el modelo consulta, filtra, cruza información iterativamente. Cada resultado informa la siguiente llamada.
En la práctica conviven — el agente usa MCP, el resto del sistema usa REST

Módulo 03

Contratos de Datos

FastMCP deriva el contrato directamente del código Python. Una sola fuente de verdad elimina el drift entre documentación, validación y esquema.

REST tradicional — contrato separado del código
1. Escribís el contrato a mano (OpenAPI / docstring)
# openapi.yaml (o docstring manual) parameters: - name: user_id pattern: "^usr_[a-z0-9]{8}$" required: true - name: days_inactive minimum: 30 maximum: 365
archivo separado puede desincronizarse
2. Luego escribís la validación (duplicado)
def delete_user(user_id, days): # validás de nuevo — ya lo dijiste arriba if not re.match(r"^usr_[a-z0-9]{8}$", user_id): raise ValueError(...) if not (30 <= days <= 365): raise ValueError(...)
lógica duplicada 3 lugares para actualizar
FastMCP — contrato derivado del código
Una sola fuente de verdad
from pydantic import BaseModel, Field class DeleteUserParams(BaseModel): user_id: str = Field( ..., pattern=r"^usr_[a-z0-9]{8}$", description="ID del usuario" ) days_inactive: int = Field( ..., ge=30, le=365, description="Días sin actividad" ) @mcp.tool() async def delete_user(p: DeleteUserParams): ...
una sola definición valida en runtime genera el JSON schema documenta al LLM
El problema del drift — qué pasa cuando cambia un requisito
REST: cambia el mínimo de 30 → 60 días
# openapi.yaml minimum: 60 ← actualizaste # handler.py if days < 30: ← olvidaste raise ... # test_docs.py assert days >= 30 ← también olvidaste
3 lugares, 2 desincronizados
FastMCP: cambia el mínimo de 30 → 60 días
class DeleteUserParams(BaseModel): days_inactive: int = Field( ..., ge=60, ← un solo cambio le=365 )
1 lugar — todo sincronizado
Lo que FastMCP genera automáticamente del modelo Pydantic
// generado automáticamente por FastMCP — nunca lo escribís a mano { "name": "delete_user", "inputSchema": { "type": "object", "properties": { "user_id": { "type": "string", "pattern": "^usr_[a-z0-9]{8}$", "description": "ID del usuario" }, "days_inactive": { "type": "integer", "minimum": 60, "maximum": 365, "description": "Días sin actividad" } }, "required": ["user_id", "days_inactive"] } }
// el modelo recibe esto y sabe exactamente cómo llamar la tool // intenta llamarla así (correcto): { "user_id": "usr_a1b2c3d4", // cumple el pattern "days_inactive": 90 // entre 60 y 365 ✓ } // intenta llamarla así (incorrecto — rechazado antes de llegar a tu fn): { "user_id": "admin_root", // no cumple el pattern ✗ "days_inactive": 5 // menor que 60 ✗ }
# el error que FastMCP devuelve al LLM (estructurado, no un crash) ValidationError: 2 validation errors for DeleteUserParams user_id String should match pattern '^usr_[a-z0-9]{8}$' Input: 'admin_root' [type=string_pattern_mismatch] days_inactive Input should be greater than or equal to 60 Input: 5 [type=greater_than_equal] # el LLM recibe este mensaje, corrige los parámetros y reintenta # tu función nunca llegó a ejecutarse
Una fuente de verdad
El modelo Pydantic es el contrato. Cambiás un ge=30 a ge=60 y todo el stack se actualiza automáticamente.
Validación previa a ejecución
Pydantic rechaza los parámetros inválidos antes de que tu función corra. Imposible que llegue un user_id malformado.
El LLM aprende del schema
El modelo ve el pattern, el minimum y el description antes de intentar la llamada. Menos errores.
Auto-corrección del agente
Si el LLM manda parámetros inválidos, recibe el ValidationError estructurado de Pydantic y puede corregirlos y reintentar.

¿Qué es la "indeterminación"?
No determinismo
El LLM puede generar parámetros distintos en dos runs idénticos. Sin validación, cada run produce un efecto diferente.
Ambigüedad semántica
"Borrar usuarios inactivos" puede significar 30 días o 30 segundos. El guard impone la definición canónica.
Hallucination de tool
El modelo puede invocar una herramienta con argumentos que no existen. El hook rechaza la llamada antes de ejecutar.
Estado derivado
Sin snapshot, una segunda ejecución del mismo prompt puede llegar a un estado diferente si el contexto cambió.

Patrones clave para escribir agentes deterministas y auditables.

Key inmutable
Usá @dataclass(frozen=True) para el contexto del agente. Una key que puede mutar es una fuente de indeterminación.
Hooks sincrónicos
Los pre-hooks deben ser síncronos y sin efectos secundarios. Un pre-hook con I/O introduce latencia no determinista.
Contratos explícitos
Cada regla de negocio tiene una sola fuente de verdad. Nunca condicionales inline en el agente; siempre rules.validate().
Checkpoints
Guardá el estado antes de cada tool destructiva. Sin checkpoint, el rollback requiere reconstruir el estado desde logs.
Dry-run por defecto
Las tools destructivas arrancan en modo dry-run. El agente necesita pasar confirm=True explícitamente.
Observabilidad
Cada ejecución genera un run_id trazable. Sin trazabilidad, debuggear comportamiento no determinista es imposible.
